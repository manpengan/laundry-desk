## 1. V2-M2 柜台核心 + 只读 AI（产品第一次可用）

**目标**：宏发能用 v2 完成一整天的真实营业——开单、打印、取衣、收款、断网也不中断。这是整个 v2 第一个"可上柜台"的版本。

### 1.1 contracts v0.2 增量（Claude 冻结）

catalog / order / payment / fulfillment(关) 四域命令与查询；打印模板变量契约（水洗唛 21 / 不干胶 22，以 M0-3 实机核清为准）；AI 工具首批投影（全部 R0–R2 只读）；离线命令白名单（`offline_allowed` 标注落地）。

### 1.2 工作分解

| 包 | 内容 | 主责 |
|---|---|---|
| B `packages/domain` | 五段计价（原价/折价/附加/加急/运费→payable）、折扣规则、票号分配（唯一·永不复用）、可取时间计算、逾期判定、状态机接入 order/garment | **Gemini**（TDD 100%） |
| C `apps/server` 业务 | catalog（服务×品类二维价目/助记码/B价）、order（开单/暂存挂单/撤销必填原因）、payment（多方式/部分付款/欠款）、取衣（整单+部分） | **Gemini** |
| C `apps/server` AI | **AI Gateway v0**：BYOK 密钥契约落地（DEK/nonce/AAD/KEK）、官方域名白名单、verifyKey、1–2 家 adapter 接入；Tool Registry 只读投影；SSE 流式 | **Codex**（安全面）+ Gemini（adapter 复用 M0-5 产出） |
| D `apps/edge-agent` | 离线队列闭环（入队/回放/幂等）、**Primary lease 落地**（M0-2 转生产）、三类打印执行（M0-3 转生产）、冲突仲裁队列、Primary 晋升流程 | **Grok** |
| E `apps/web` | 工作台、开单页（键盘流 ≤15 秒）、取衣页（含部分取衣）、顾客页（散客）、设置（价目/字典/打印）、**AI 抽屉（只读）**、AI 接入页最小闭环、连接状态条 | **Grok** |
| F `tools/` | migrate-v1 **实跑**：宏发 v1 数据全量试迁 + 差异报告（order_items→order_lines+garments 拆件补条码） | **Gemini** |

### 1.3 M2 门禁（Claude 验收 + Codex 二审安全项）

承接 M1 全部门禁，新增：

1. **开单速度**：熟手键盘流实测 ≤ 15 秒/单（脚本化度量，种子价目下）
2. **断网演练**：拔网线 → 开单 → 打印 → 取衣（Primary 终端）→ 恢复 → 同步报告正确、无重复入账
3. **部分取衣**：一单多件勾选取走，剩余件状态与票号正确延续
4. **三类打印实机**：三台各出正确样张（依赖 M0-3 修完 + 实机日）
5. **AI 只读铁律**：契约测试断言——M2 全部 AI 工具在任何输入下**不产生任何数据库写**（红队用例覆盖诱导写入）
6. **BYOK 安全**：key 不出现在任何响应/日志/错误/AI 上下文；自定义 base_url 在 M2 **默认关闭**（M5 才开授权 UI）
7. **迁移**：宏发数据试迁差异报告为零丢失（金额/件数/客户数三项核对）

### 1.4 关键风险

- **件级拆分是 v1→v2 最大不可逆点**：迁移器补发条码的规则必须在 M2 早期定稿并试跑，晚了会拖到 M3。
- 开单 15 秒目标与"逐件录入属性"天然冲突——UI spec 已给方案（数量在加入品类时输入，属性逐件在抽屉补），M2 首周就要做可用性验证。

## 2. V2-M3 会员储值 + 顾客触达 + 开放 R3 写

**目标**：储值现金流跑通（产品化的第一动机），催取从"人工打电话"变成"名单+一键触达"。

### 2.1 contracts v0.3 增量

membership 全域（卡类规则/发卡/充值/次卡/积分/挂失/合并/退卡）；notification（12 触发节点/模板/降级名单）；保管费规则；**首批 R3 写工具**（草稿生成 + 确认发送）。

### 2.2 工作分解

| 包 | 内容 | 主责 |
|---|---|---|
| B domain | 会员余额派生（ledger 唯一真源）、金额三元组（充值/赠送/实收）、先扣本金/赠额策略、次卡扣减、积分规则、保管费计算 | **Gemini** |
| C server | membership 全量服务；notification adapter（微信模板消息 + 短信）+ **降级名单**（通道未开通时生成人工名单，这是宏发现状的刚需）；催取三档 | **Gemini** |
| C server 安全 | **R3 写通道首次真实启用**：确认卡 WYSIWYS 全链路（canonical args 冻结/args_hash/实体版本/nonce/5 分钟有效期）；批量阈值触发 R3→R4 升级 | **Codex** |
| E web | 会员全套页面（办卡/充值/升级/挂失/合并/退卡）、催取工作台、通知设置、**确认卡 UI**（含倒计时与作废态） | **Grok** |

### 2.3 M3 门禁

1. **余额铁律**：契约测试断言任何路径都不能直接 UPDATE 余额，只能追加 ledger
2. **确认卡 WYSIWYS**：换参作废、过期不可执行、canonical 冻结、step-up 不可自核（E2E 断言）
3. **降级路径**：通道未开通时，催取主按钮变为"生成名单并复制号码"，绝不假装已发送
4. 通知全量落 `notification_log`（含失败与费用）

### 2.4 关键风险

微信模板消息与短信需要**资质与账号**（顺科的实况是三项全未开通）——M3 开工前 manpengan 需启动资质申请，否则只能交付降级路径，通知链路无法端到端验收。

## 3. V2-M4 账务双口径 + 老板端 + 备份还原

**目标**：老板不在店里也能看数；账目经得起对账；备份**真的能还原**。

### 3.1 contracts v0.4 增量

accounting（科目树/日结/月结/职员/交班）；reporting 指标集；backup/restore 命令；老板端只读查询集。

### 3.2 工作分解

| 包 | 内容 | 主责 |
|---|---|---|
| B domain | **业绩收入 ≠ 真实收入**双口径归集（排除储值款、含刷卡消费）、科目树归并、交班差额计算 | **Gemini** |
| C server | accounting 服务、reporting 聚合、备份加密上传 + **还原演练命令**、导出（Excel/CSV + 照片打包） | **Gemini** |
| C server AI | 分析类 AI：经营问答深化（带来源与筛选条件）、对账异常解释、每日/每周经营周报（定时生成，非自动执行写操作） | **Codex** + Gemini |
| E web | 账目四页签 + 交班页（应有/实点现金差额高亮）、**老板端 H5**（指标卡/趋势/AI 问答/周报）、备份还原 UI | **Grok** |

### 3.3 M4 门禁

1. **还原演练是功能不是口号**：一键把最新备份恢复到影子库 + 校验查询通过，`restored_test_at` 落库；CI 每周自动演练
2. **对账**：用已知的若干笔单反推报表数字，双口径各自自洽
3. 导出全量数据可用（数据归属承诺的兑现）

## 4. V2-M5 AI 完整面

**目标**：AI 从"读+确认写"升级到"有边界自动化"，BYOK 全厂商可用。

### 4.1 contracts v0.5 增量

`automation_policies` 全域；`approval_requests`（异步）；用量计量与限额；自定义网关授权。

### 4.2 工作分解

| 包 | 内容 | 主责 |
|---|---|---|
| C server 安全 | **有边界自动化策略**（仅 ≤R3、对象过滤、每日次数/金额/时间窗限额、owner 审批生效、一键暂停、硬编码红线）；**异步审批中心**（R4 流）；自定义网关 SSRF 全套硬化（DNS 校验/防 rebinding/逐跳重定向/独立 egress 代理） | **Codex** |
| C server | BYOK 全矩阵（三 adapter × 全部注册表厂商）、用量计量 `ai_usage_daily`、org 月度限额熔断 | **Gemini** |
| E web | AI 接入页全矩阵 + 用量看板、自动化策略管理 UI（含红线提示）、审批中心页 | **Grok** |

### 4.3 M5 门禁（本期安全权重最高）

1. **红线不可配置**：契约测试断言退款/免单/余额/权限/密钥/备份恢复在任何策略配置下都不可自动执行
2. **SSRF**：负向用例覆盖 loopback/RFC1918/link-local/metadata/DNS rebinding/重定向逃逸，全部拒绝
3. 限额熔断真实生效（超限即停并通知 owner）
4. 多模型兼容矩阵进 CI

## 5. V2-M6 四子期（各自独立门禁，不合并交付）

| 子期 | 范围 | 主责 | 前置 |
|---|---|---|---|
| **6.1 视觉 AI** | 拍照辅助开单（出候选 chips，**不自动定价**）、掉标找衣（以图搜图，对标洗衣管家唯一验证过的行业 AI 卖点） | Gemini(模型侧) + Grok(端侧) | M2 件级照片链路 |
| **6.2 小程序** | 顾客端：件级进度时间线、在线下单预约、钱包（余额/次卡/券包）、微信支付 | **Grok** | M3 会员 + 微信资质 |
| **6.3 工厂协同** | fulfillment 全量（送洗/上挂/格位）、店厂批次四节点扫码交接、工序看板、双向对账 | Gemini + Grok | M2 状态机（`fulfillment` flag 打开） |
| **6.4 取送与营销** | 聚合运力 adapter（闪送/顺丰同城/达达）、运费入单、轨迹回传；优惠券/满减/套餐卡/单层推荐；美团抖音团购核销 | Gemini | 商务门槛（企业认证） |

**排序建议**：6.1 最先（复用已有链路、差异化最高、无外部门槛）→ 6.3（宏发若外送工厂则刚需）→ 6.2（等微信资质）→ 6.4（等商务/运力签约）。四者**无技术互锁**，可按资质与商务进度灵活调序。

### 6.2 每期固定节奏

```text
期初：Claude 冻结 contracts 增量并打 tag → 各 AI 按目录所有权开工
期中：周五集成日全量 E2E；Codex 对高危 PR 二审
```

### 6.3 铁律（M0 教训固化）

1. **证据必须可复现且不得删改输出标识**——mock 输出冒充实测是门禁红线（M0-5 判例）
2. **不得越期开工**——M1+ 内容一律等 contracts 冻结信号（M0 违规判例）
3. **不改他人目录**，跨模块需求走 contracts 提 issue
4. 每个 PR 独立分支、独立 CI 绿灯、不夹带无关文件
