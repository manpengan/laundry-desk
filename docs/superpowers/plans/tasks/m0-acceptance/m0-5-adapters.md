# M0-5 验收单：三 adapter × 模型工具调用兼容矩阵

> 主责：**Gemini**　验收：Claude　产出：`tools/spikes/m0-5-adapters/`
> 依据：ADR-06（后果条款：M0 含兼容矩阵）、架构 §9.2/§9.3、任务书 gemini §2
> 公共规则见 [README](README.md)。**API key 只走环境变量，仓库/日志/findings 零明文。**

## 1. 目标

AI Operator Runtime 假设"一套统一 tool-use 循环适配三协议"。本 spike 用同一用例实测三 adapter × 代表模型，证明闭环成立并把方言差异记录成表——这是 M1 Tool Registry 与 M2 BYOK 最小闭环的地基。

## 2. 步骤

1. 定义最小统一内部消息格式：`text / tool_use / tool_result / image` 块（image 仅定义类型占位，不实测视觉）。
2. 三 adapter：
   - `anthropic`：官方 TS SDK，Messages API；
   - `openai-compat`：自定 base_url，从 DeepSeek/Qwen/GLM/Kimi/豆包/Grok 中**任选 ≥2 家实测**；
   - `gemini`：**原生 API**（其 OpenAI 兼容端点工具调用阉割，不得使用）。
3. 同一测试脚本对每个模型跑四段用例：
   a. 单工具闭环：查询 → `tool_use` → 回填 `tool_result` → 模型总结 → 正常结束；
   b. 并行工具：一轮返回两个工具调用（不支持并行的模型记录其串行降级行为）；
   c. 流式：stream 事件逐块归一到统一格式，工具调用在流式下同样成立；
   d. 参数解析：工具参数一律 `JSON.parse` + Zod 校验——**禁止字符串匹配/正则抠参**。
4. 稳定性判定口径（本单裁定）：同一用例**连续 3 次全绿 = "稳定"**；偶发失败记录现象与频率，不隐藏。
5. 方言差异表维度：工具定义格式 / 调用块形态 / 流式事件类型 / 结束条件 / 并行语义 / token 用量字段。

## 3. 通过标准（逐条判定）

- [ ] 每 adapter **≥1 款模型稳定完成闭环**（连续 3 次：查询→工具→回填→结束 全绿）。
- [ ] `openai-compat` 实测 ≥2 家厂商（记录 base_url 域名与型号；key 本身不出现）。
- [ ] 并行工具行为逐模型记录（支持 / 不支持 + 实际表现）。
- [ ] 流式模式下工具调用闭环成立（如某模型只能非流式，记录为方言）。
- [ ] 参数解析代码零字符串匹配（code walk 可证）；坏 JSON/缺字段有明确失败路径。
- [ ] 方言差异矩阵表完整：六维 × 全部实测模型。
- [ ] 仓库与全部证据零 key 明文（提交前 `git log -p` 自查一遍并在 README 声明）。

## 4. 证据格式

- `tools/spikes/m0-5-adapters/README.md`：所需环境变量清单（只列变量名）、运行方式、结果摘要。
- `evidence/`：逐模型运行 transcript（脱敏后的请求/响应关键片段 + 断言结果）、连续 3 次运行记录。
- 统一消息格式类型定义独立成文件——即 M1 契约里 AI 消息格式的实测底稿。
- findings `## M0-5` 小节：结论行 + 方言差异矩阵表 + 各厂商坑列表。

## 5. 不通过 / 需改设计

某 adapter 全部候选模型都无法稳定闭环 → 报「需改设计」，Claude 评估该协议线是否降级/推迟（影响 §9.2 与 M2 BYOK 范围）并起草 ADR。个别模型不稳定不算不通过——记录进矩阵即可，注册表是配置驱动的。
