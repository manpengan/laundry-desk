# T5 规格：跨租户负向用例 → CI 门禁

> 起草：Claude（设计与门禁）　日期：2026-07-20　T5 交付物
> 执行：**Codex**（随 C2 RLS 接入交付）　验收：Claude
> 依据：ADR-02 第 6/10 条、架构 §4；底稿：M0-1 spike（`tools/spikes/m0-1-rls/`，已判**通过**）

## 1. 目标与转正差量

M0-1 已实证 **SQL 层**五类旁路全部 fail-closed，这部分**直接转正、不重新设计**。但 ADR-02 第 6 条要求的门禁是**三条路径**——API、队列任务、报表导出——M0-1 一条都没覆盖（spike 直连 psql）。本规格的实质内容就是补齐这三条应用层路径：**RLS 兜底住了数据库，门禁要证明应用层没在数据库之前就把数据漏出去。**

## 2. 用例分层

### L0 SQL 层（M0-1 转正，5 类 × 读写两侧）

原样移植 `tools/spikes/m0-1-rls/sql/acceptance.sql` 的判定逻辑：GUC 未设置 / GUC 空值 / 事务回滚残留 / 连接池复用串租户 / worker 漏注入，每类读侧断言 0 行、写侧断言 RLS 违例。**保留其 `RAISE EXCEPTION` 判负写法**（断言真会失败，M0 教训 3）。

### L1 API 路径（新增）

- 租户 A 的有效会话请求租户 B 的资源 ID → 404/403，**响应体不得泄漏存在性**（不得出现"无权访问该门店订单"一类可用于探测的措辞）。
- 客户端在请求体/查询串/请求头中**自报** `org_id`/`store_id` 与会话不符 → 自报值被忽略（以会话为准），不得据以查询（ADR-02 第 10 条、架构 §4）。
- 同 org 跨 store：A 店员工请求 B 店订单 → 拒绝（**二元隔离防不住这条，是三元组合键的核心价值**）。
- 会话缺失/过期时命令入口 → 拒绝，且**不落入无 GUC 的数据库查询**。

### L2 队列 worker 路径（新增）

- worker 处理任务时**未注入** GUC → 查询 0 行且任务判失败，**不得静默跳过**（静默跳过 = 数据处理丢失，比报错更坏）。
- 任务 payload 自带 `org_id/store_id` 与任务归属不符 → 以服务端归属为准，payload 值不生效。
- 同一 worker 连接连续处理 A、B 两租户任务 → 无上下文残留（对应 L0 的连接池复用场景在 worker 侧的等价物）。

### L3 报表/导出路径（新增）

- 导出接口在租户 A 会话下产出的文件**逐行**属于 A（断言行级，不是断言行数）。
- 跨 org 聚合查询（如全局统计）在应用角色下执行 → 只覆盖本租户；若确需跨租户统计，必须走**独立的、有审计的**路径，不得由普通会话触发。

### L4 反旁路（新增，防门禁自身被绕过）

- **`maintenance_policy` 不得成为旁路**：M0-6 复验发现 owner 无策略覆盖会被 FORCE RLS 全量拒绝（迁移/回填/seed 全挂），故 init.sql 补了 maintenance_policy。门禁必须断言：应用角色 `laundry_app` **无法**借该策略读到跨租户行。
- 应用连接角色 `rolbypassrls = false` 且非表所有者（运行时断言，不是文档承诺）。
- 所有租户表 `relrowsecurity` 与 `relforcerowsecurity` 均为 true——**新增表必须自动纳入**：门禁枚举 schema 内全部租户表，漏配即红（防"新表忘了开 RLS"这一最可能的回归）。
- 非法但非空的 GUC 值（如畸形 UUID）→ 仍 fail-closed；**不得**因转换异常被上层 catch 后当作"无租户"放行（M0-1 偏差条款：M1 只注入已校验的服务端 UUID）。

## 3. 通过标准

- [ ] L0–L4 全部用例在 CI 中实跑通过；CI 内 PG 服务与 `tools/compose/` 版本一致。
- [ ] **每条断言可失败**：提交前逐条人为破坏验证会红（如临时删一条策略、临时信任客户端自报值），并附破坏验证记录。**不接受 `|| echo PASS` 式恒真写法。**
- [ ] L4 的"全部租户表枚举"是动态查 `pg_class` 得来，不是硬编码表名列表（否则新增表天然逃逸）。
- [ ] 门禁在两条 CI 线中的**归属明确**（建议 V2 Foundation 线），失败即阻断合并。
- [ ] 用例与实现分离：门禁读的是公开 API/worker 入口，不 import 内部实现细节（否则重构即误红）。
- [ ] 单文件 ≤400 行、函数 ≤50 行；测试数据全虚构（手机号 `13800000xxx` 段）。

## 4. 证据格式

- CI 运行链接 + 全绿截图或输出。
- **破坏验证记录**：逐条列"破坏了什么 → 哪条用例红了"（这是本门禁最重要的证据，证明它不是恒真）。
- 用例清单与本规格 §2 的映射表（L0–L4 逐条 → 测试用例名），缺项标注理由。

## 5. 不通过

L1–L3 任一路径未覆盖 = 不通过（只转正 L0 等于没做本项）。断言无法失败 = 不通过。L4 的表枚举若硬编码 = 不通过。
