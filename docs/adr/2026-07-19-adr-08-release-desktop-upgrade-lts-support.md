# ADR-08: 发布、桌面升级、LTS 与技术支持

- 日期：2026-07-19　状态：Proposed　父文档：[总 RFC](2026-07-19-v2-productization-and-ai.md)
- 详设：架构 §13（自 ADR-07 拆出，三审建议）

## 决策

### 版本、渠道与兼容

1. 全线 SemVer；**兼容单位 = `packages/contracts` 协议 major**（不兼容变化只能进 major；"上一个产品 release"不是兼容单位）。server 同时支持当前与上一 major，废弃期 ≥ 1 个里程碑 + Deprecation 响应头；双 major 兼容测试进 CI。
2. 渠道：beta / stable（SaaS）+ **LTS（私有化，≥ 24 个月）**；服务端按 org 分批灰度。
3. **支持矩阵**随每次 release 发布：Server × Web × Edge × contracts major × Edge 本地 schema 五维兼容表——升级/回滚的唯一判定依据。
4. 服务端数据库迁移 expand → migrate → contract 三段式；升级前自动备份点；自托管逐 minor 升级 + 迁移门。

### 桌面（Edge）升级状态机

5. **A/B 双槽程序包** + 启动健康检查（硬件自检/本地库可打开/server 握手），失败自动回原槽。
6. 升级前置：离线队列已清空；窗口内**不签发新的 Primary lease**；创建本地库加密快照；队列未清空不得安装。
7. Edge 本地库（SQLCipher）同样三段式迁移；**离线队列信封独立版本化**；server 保留旧版队列回放接口，保留期长于 UI 双 major 兼容期。
8. **回滚判定**：旧版本能读当前本地 schema（contract 未执行）才允许自动回滚；否则进入**恢复模式**（只打印+只读）等待前滚修复，禁止盲目降级。autoUpdater 只换二进制，数据兼容由本状态机负责。
9. 更新包、manifest、hash、回滚目标全部签名；带**最低安全版本**（anti-rollback）。
10. **LTS ≠ 冻结运行时**：Electron 官方仅支持最新三个 stable major——LTS 保持业务功能稳定，持续跟进 Electron/Chromium 安全版本。

### 技术支持

11. 诊断包一键导出（脱敏）；产品内 R0 规程/排查助手为一线自助。
12. **远程协助**：店主重新认证 + 明示授权、会话中可实时撤销、连接从门店出站、支持人员 MFA、**命令级只读白名单**（禁裸库/密钥/任意 shell）、1h 过期、全程审计。
13. 支持窗口：stable ≥ 12 个月 / LTS ≥ 24 个月；安全补丁回移所有受支持版本；响应分级 P1 4h / P2 1 工作日（商务条款细化）。

### 交付形态

14. **桌面为主、Web 次之**（用户裁定）：断网冷启动的桌面壳是主交付物；前期开发以本地 web 服务（单机模式）做测试适配。

## 理由

三审 P0：draft3 的"失败自动回滚"没有处理"新版已迁移本地库/写入新版队列后回滚旧二进制"的情况；升级期间签发 lease 会造成租约与程序版本不一致；无 anti-rollback 则攻击者可降级到含漏洞版本。

## 否决的备选

- 依赖 Electron autoUpdater 自带机制解决一切（它不管数据兼容）。
- 单槽就地覆盖升级（失败即砖）。
- LTS 冻结 Electron 版本（Chromium 漏洞窗口不可接受）。

## 后果

- M0 验证矩阵含 A/B 槽升级+回滚演练（本地库快照恢复）；M1 冻结队列信封版本字段与支持矩阵格式。
- 发布工程（签名流水线、双槽打包）是 `apps/edge-agent` 所有者的交付物之一。
