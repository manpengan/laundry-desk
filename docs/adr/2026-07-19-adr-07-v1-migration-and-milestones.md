# ADR-07: v1→v2 迁移与里程碑（方案 B）

- 日期：2026-07-19　状态：**Accepted**（2026-07-19 定点复核通过，批量签署）　父文档：[总 RFC](2026-07-19-v2-productization-and-ai.md)
- 详设：架构 §14

## 决策

1. 实施路线采用**方案 B：AI-first 垂直切片**——M1 建命令总线/权限/审批/工具注册，M2 上线只读 AI，随里程碑逐步开放安全写操作。
2. 里程碑：**V2-M0 技术验证**（RLS 性能与旁路负向测试、三类打印机实机、离线冲突演练含 Primary 规则、**Windows 浏览器↔Edge WSS/LNA 实测**、三 adapter 工具调用兼容矩阵）→ M1 基座（identity/RLS 含租户矩阵冻结/总线含元数据四字段/审计同事务/Tool Registry/Policy v0 **含 R4 同步 step-up**/Edge v0）→ M2 柜台核心+只读 AI+BYOK 最小闭环（R4 命令经 step-up 执行）→ M3 会员/触达/开放 R3 → M4 账务/老板端/备份+分析类 AI → M5 AI 完整面（BYOK 全矩阵/**异步**审批中心/有边界自动化）→ **M6 拆四子期**：6.1 视觉 AI、6.2 小程序、6.3 工厂协同、6.4 取送+营销。
3. 首批 AI 功能集（映射入上述里程碑）：全局操作助手、带来源的经营问答、滞留分析+催取确认发送、规程/排查助手、对账异常解释、拍照辅助开单（不定价）、掉标找衣、每日/每周经营报告。
4. **v1→v2 迁移器**（`tools/migrate-v1`）：v1 SQLite → v2 PG；`order_items`（品类×数量）→ `order_lines` + 按 qty 拆生成 `garments` 并补发条码；customers/photos/settings 直迁；缺失 `expected_pickup_date` 按规则补算。宏发为首个迁移案例，切换窗口双写观察细案在 M1 内产出。
5. 宏发 v1 的 M1–M4 收口流程不因 v2 打乱。
6. **桌面为主、Web 次之（三审后用户裁定）**：桌面壳（含断网冷启动）是主要交付形态，每里程碑含桌面交付物；**前期开发以本地 web 服务（单机模式：server+PG 本地 compose）做测试适配**，同一套代码后续上云（架构 §2/§13.3）。
7. 发布、桌面升级、LTS 与技术支持独立成 **[ADR-08](2026-07-19-adr-08-release-desktop-upgrade-lts-support.md)**（三审建议，避免本 ADR 大杂烩）；M0 验证矩阵新增：Primary lease 时序、桌面断网冷启动、A/B 槽升级回滚演练（架构 §14）。

## 理由

- 方案 A（AI 后置 M5）与 AI-first 目标冲突且造成接口返工（二审）。
- 方案 C（现有 v1 直接套 AI）不可作生产架构：v1 web 服务是通用 channel+payload 直通调用（`src/main/server.ts`），无会话、多租户与命令风险边界，仅适合一次性原型。
- M6 原来四块（小程序/工厂/取送营销/视觉）体量互不相关，合并交付会拖垮门禁，拆为独立子期。

## 否决的备选

方案 A、方案 C（如上）；一步到位大爆炸式重写（无 M0 验证与垂直切片，风险不可控）。

## 后果

- 每期沿用既有门禁流程（Claude 验收、关键节点 Codex 二审），新增门禁项：跨租户负向测试、AI 红队用例、离线演练、确认卡 WYSIWYS 断言。
