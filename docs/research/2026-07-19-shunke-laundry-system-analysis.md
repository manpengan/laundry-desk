> 分析日期：2026-07-19

> 分析方式：只读文件盘点、PE/依赖识别、资源图审阅、备份头与运行日志审阅、与 `laundry-desk` 当前源码对照  
> 重要边界：本机没有 Wine 或 Windows 虚拟机，本轮没有执行任何未知 `.exe`，没有恢复或查询客户数据库，也没有展示客户姓名、电话、订单等业务数据。

## 1. 执行摘要

顺科不是一个简单的“开单软件”，而是一套已经长期投入门店使用的 Windows 洗衣店综合管理系统。依据主程序、ActiveX 控件、图片目录、音频提示、SQL Server 备份和云上传日志，可以较有把握地判断，它覆盖了以下主线：

1. 前台收衣、衣物/服务项目选择、定价、优惠、收款、打印、条码和拍照。
2. 取衣查询、票据或条码检索、补打以及可能的异常/遗失处理。
3. 客户与会员管理，包括会员卡、储值/消费、促销、礼品或积分兑换等经营功能。
4. 现金、卡、微信或其他在线方式的收款记录，以及营业/财务统计图表。
5. 操作员登录、系统设置、数据查询、打印和软件更新。
6. 短信、微信入口、语音提示、摄像头、条码、IC 卡读写器和手写输入等柜台外设能力。
7. SQL Server 本地数据库、频繁本地备份、RAR 压缩和阿里云 OSS 异地上传。

它的优势是功能密度、传统门店硬件适配和长期实战积累；主要问题是底层技术极老、安装维护脆弱、数据安全边界不清、备份无限增长，以及现代 Web/移动访问能力不足。

## 4. 推测的核心工作流

以下流程由功能证据组合还原，并非实机逐步点击结果。

### 4.1 收衣

1. 操作员登录。
2. 搜索客户或读取会员卡，新客户建立档案。
3. 选择衣物/服务项目和价格，应用会员价、折扣或促销。
4. 录入衣物备注，并可拍摄衣物照片。
5. 选择现金、卡、微信/在线等收款方式。
6. 生成条码或标签，打印收衣单/标签。
7. 发送短信或微信类提醒，并播放语音提示。

### 4.2 取衣

1. 通过条码、客户、会员卡或单据条件查询。
2. 打开衣物/订单详情，核对未结金额和状态。
3. 收尾款或完成卡消费。
4. 确认取衣、打印取衣凭证；异常场景可能支持作废、删除、遗失处理或补打。

### 4.3 会员经营

1. 建立客户与会员卡。
2. 充值/消费、余额或卡状态管理。
3. 应用会员价格、折扣、优惠券或促销。
4. 进行积分/礼品兑换，并通过短信/微信触达。

### 4.4 经营管理

1. 按操作员、客户、时间或业务类型查询。
2. 查看现金、储值、消费和营业统计。
3. 通过饼图/动态图表展示经营结构。
4. 打印或可能导出 Excel。

其优势是入口密集、老员工操作路径短；缺点是信息层级可能拥挤、分辨率适配弱、现代可及性和响应式布局不足。

## 7. 技术与运维风险

### 7.1 高风险

1. **SQL Server 2000 已严重过时**：数据库版本和网络协议都缺乏现代安全维护。如果端口暴露到不可信网络，风险很高。
2. **VB6/OCX 部署脆弱**：依赖 1998–2002 年的多个系统控件，并需要管理员复制/注册 OCX。
3. **主程序无数字签名**：无法通过 Authenticode 验证发布者和更新链完整性。
4. **备份隐私风险**：597 份原始业务备份位于未加密 FAT32 介质，并上传第三方云存储；访问控制和加密状态未知。
5. **恢复能力未证实**：备份数量很多，但没有恢复演练、校验或损坏检测证据。

### 7.2 中风险

1. 主程序经过打包/保护，缺少源代码时维护和迁移成本极高。
2. 数据库、打印机、摄像头、读卡器、短信、手写输入均存在驱动/控件版本耦合。
3. 云备份日志出现过 DNS 失败，监控、自动重试和告警机制未知。
4. 备份没有保留上限，数据目录会持续增长。
5. 登录界面存在不等于安全；密码哈希、角色权限、操作审计均未验证。
6. 微信图标存在不等于直接支付接口；可能只是收款方式标记、公众号或消息入口。

## 9. 对 `laundry-desk` 的产品建议

### P0：当前收口后优先补齐

1. 完成 M4 登录、权限、审计和短信密钥保护。
2. 增加真正的异地备份：端到端加密、校验、30/90 天分层保留、恢复演练。
3. 把条码/标签打印纳入打印驱动抽象，支持补打和打印失败队列。
4. 建立 Windows 外设验证矩阵：58mm 打印机、扫码枪、摄像头、不同 DPI/分辨率。

### P1：高价值柜台体验

1. 可关闭的语音提示与错误提示音。
2. 快捷键、大按钮和触屏优化。
3. 衣物照片分类、异常/遗失记录和照片生命周期管理。
4. 更完整的收款方式、欠款、日结和复购分析。

### P2：v1 后经营能力

1. 会员储值、积分、优惠券、礼品兑换。
2. IC/会员卡读写器适配。
3. 微信公众号/小程序或支付网关，但必须独立做账务对账和安全设计。
4. 取送/配送或送厂流程；需先确认真实门店是否需要。

### 不建议照搬

1. 不采用 SQL Server 2000、VB6/OCX 或管理员注册系统控件的方案。
2. 不把所有备份永久堆在同一个未加密目录。
3. 不在业务主程序中硬绑定打印机、读卡器、摄像头和短信厂商。
4. 不把“有图标/有入口”当作安全、审计或可恢复性已经完成。
5. 不因为顺科功能多而打乱当前 M1–M3 收口和 M4 安全边界。

需要 Windows 隔离环境和脱敏数据副本才能进一步确认：

1. 完整菜单树、每个图标对应的真实模块名称。
2. 会员储值、积分、礼品、优惠券的精确规则。
3. 微信入口究竟是支付、收款方式、公众号还是通知。
4. 条码是订单号、衣物标签、取衣凭证还是多种用途。
5. 是否支持多门店、多台客户端、洗衣工厂交接或配送。
6. 员工角色、密码存储、审计日志和删除权限。
7. SQL 备份是否可在现有环境完整恢复，云端备份是否仍健康。
8. Excel 的实际导入/导出范围。
9. 短信供应商、失败重试和费用统计。
10. 退款、撤单、欠款、补打、遗失赔偿等异常流程。

## 11. 下一阶段验证方案

如需把本报告从“静态高置信分析”升级为“逐页面实测”，建议：

1. 先复制样本到隔离 Windows 虚拟机，不直接从 USB 原盘运行。
2. 禁止外网并创建快照，避免自动更新、云上传或修改原始数据库。
3. 使用脱敏/空数据库启动，记录完整菜单、窗口和操作路径。
4. 分别测试登录、建客、收衣、拍照、收款、打印、取衣、报表、备份和恢复。
5. 抓取外联域名和端口，但不记录账号、短信密钥或客户数据。
6. 最后形成页面级功能矩阵、操作视频和 `laundry-desk` 差距 backlog。
