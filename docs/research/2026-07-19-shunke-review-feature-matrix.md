## 0. 复核总结论

### 0.1 两份报告的准确性

**两份报告在"功能有没有"这一层基本准确**，收衣页字段、金额结构、菜单树、会员操作清单、打印三件套等核心描述逐条对得上截图。截图报告第 25 行主动声明"配置页存在 ≠ 已启用"，这个方法论是对的。

**但两份报告都停在了"软件能做什么"，没有回答"这家店实际在用什么"。** 这是本轮复核最重要的发现，也是唯一会实质改变 laundry-desk 优先级判断的发现。

### 0.4 两份报告都没提到的新证据

- **`指定时间段内业绩/详单`（IMG_2331）是最完整的财务口径页**：32 个明细维度 + 10 个汇总维度，含 `代收点收`、`菜鸟驿站`、`美团订单` 等渠道。
- **顺科明确区分"业绩收入"与"真实收入"**（IMG_2343 原文）：`业绩收入≠真实收入，因为排除了储值款，包含了刷卡消费`。这是储值制洗衣店的会计口径难点，laundry-desk 若将来做储值必须照抄这个区分。
- **散客可按客户关闭流程**（IMG_2319）：`不打印取衣单` / `不分配挂点` / `不打印水洗唛` 三个客户级豁免开关。
- **三台打印机全部实配在用**（IMG_2314/2315/2316）：小票 `XP-58`（USB/58mm）、水洗唛 `DASCOM DL-206`（有切刀）、不干胶 `Gprinter GP-3120TL`，模板已填变量、打印时机已设。这是顺科**唯一硬件层真正跑起来**的部分。
- **管理员数据工具风险敞口大**（IMG_2322）：15 个按钮（含 `强制删除`、`自查垃圾数据并删除`、多种"合并"）外观一致、无逐项说明、无二次确认，只有窗口顶部一条统一红字警告。
- **1067 件长期未取 = 全部收衣 14451 件的 7.4%**（IMG_2348 + IMG_2327），最早滞留件收衣日期为 2025-05。这是这家店真实存在的经营问题，而催取和保管费两个对应工具一个在用（催取）、一个没开（保管费）。

## 1. 核心业务闭环

### 1.1 软件设计的闭环（顺科提供的能力）

```text
闭环 A｜散客收衣
  手机号/姓名找人 → 选服务大类(11) → 选品类(12 分类) → 逐件填 颜色/品牌/备注/附加/条码/附件/款式/分区/数量
  → 原价·折价·附加·加急·运费 五段计价 → 核价总额 → 选付款方式
  → 打印 取衣联/存根 + 水洗唛 + 不干胶 → 短信/微信回执 → 进入生产

闭环 B｜会员消费
  读卡/手机号 → 载入 余额·赠额·积分·鞋次卡·衣次卡·10 类分品类折扣
  → 按会员规则计价 → 余额/赠额/次卡/现金结算 → 更新积分与消费历史 → 回执

闭环 C｜生产交付
  已收衣 → 送洗(至工厂/至本店) → 完工上挂(登记分区+挂点，语音播报，打不干胶)
  → 待取 → 按 姓名/电话/票单号/条码/挂点 查找 → 整单取衣 / 部分取衣
  → 补款·收保管费·返工·挂失·回送

闭环 D｜长期未取
  库存待取排查(按收衣日期/单号区间/未付款；按 已收衣·已送洗·已上挂 筛状态)
  → 按单 或 按顾客 汇总 → 三档催取(单个 / 按订单批量 / 按顾客批量)
  → 通道策略(优先微信、未关注转短信 / 全部短信) → 逾期自动计收保管费 → 取衣收款

闭环 E｜门店结账
  当班 收衣·取衣·充值·支出·补收欠款 → 按支付渠道汇总
  → 核对 预留备用金 与 钱箱现金 → 交班结账 → 打印/短信/微信发老板 → 留存日结·月结·历史详单

闭环 F｜线上经营（未开通）
  公众号关注绑定 → 在线注册/下单/充值/欠款支付/红包 → 推广员三级分销 → 金币兑换余额
```

### 1.2 门店实际跑通的闭环（只有两条半）

```text
实跑 1｜收衣 → 取衣（两态）
  找散客 → 选服务+品类 → 逐件录入 → 核价 → 微信/现金/未付款
  → 打小票 + 水洗唛 + 不干胶 → 【无送洗、无上挂、无挂点、无回执短信】
  → 客人来了按 票单号/条码/姓名/电话 查 → 整单或部分取衣 → 补欠款

实跑 2｜欠款回收
  收衣时选 未付款（已开启）→ 未付款统计 126 单 / 5811 元
  → 按客户或卡类查 → 催付 → 取衣时 回收欠款（月结显示现金 800 元全部来自此项）

实跑 0.5｜库存催取
  库存待取 1067 件 → 可查、可排名、可导出、可打印
  → 催取按钮存在且模板已填，但短信/微信通道均未开通 → 实际只能靠人工打电话
```

### 1.3 差异的含义

顺科把"生产流转 + 会员经营 + 线上营销"做成了产品主线，但这家店只用了"开单 + 打印 + 取件 + 收欠款"。

**对 laundry-desk 的直接含义：不要按顺科的功能清单排优先级，要按这家店实际跑通的闭环排。** 生产流转、会员储值、微信生态这三块在顺科是"配置齐全但零使用"，照搬会造出同样没人用的功能。

### 3.3 缺口的三句话总结

1. **真正致命的是四件事**：单件衣物明细、件级条码、水洗唛/不干胶标签、部分取衣。这四件互相咬合——没有件级实体，就没有条码；没有条码，就没有标签；没有标签和件级状态，就没有部分取衣。顺科每天靠这一串在跑。
2. **第二梯队是计价与补打**：加急费/附加费/运费缺失，折扣是坏的，重新打印和撤销没有。
3. **顺科看着最"厉害"的那部分（会员储值、生产流转、微信生态、分销、保管费）在这家店全是零使用**，照抄只会复制一堆没人点的按钮。

## 4. 功能优先级建议

排序依据：**这家店实际在跑的闭环 > 会造成货物/金额错误的缺口 > 效率 > 增长**。仅为产品优先级，不含实现方案。

### P0 — 不做就无法替代顺科

1. **单件衣物明细模型**：一单多件，逐件记录 服务/名称/颜色/品牌/备注（瑕疵）/附件/款式/数量/原价/折价。这是整个系统的最小业务对象，其他 P0 都依赖它。
2. **件级条码 + 水洗唛标签打印**：顺科三台打印机里，水洗唛是唯一"衣物离店进厂后还能认回来"的凭据。缺它等于缺失物防线。
3. **不干胶标签打印**（取衣/交付环节）。
4. **部分取衣**：一单多件时客人只取一部分是常态，必须有件级取走状态。
5. **修复折扣链路**：当前折扣 > 0 必定下单失败，属于线上事故级缺陷。
6. **补齐计价段**：加急费、附加费、运费。顺科每单都在算这三项。
7. **修复逾期链路**：`expected_pickup_date` 必须在收件时写入；顺科的"可取日期"是收件页自动算出来的（收到后 N 天 M 时），可直接对齐。
8. **重新打印**：小票 / 水洗唛（整单 + 单件）/ 不干胶，至少三类补打。
9. **订单撤销 + 必填撤销原因**：顺科把原因设为强制字段，是合理的责任留痕。
10. **M4 登录 + 权限 + 审计绑全**：门禁硬要求，且顺科的 85 项权限说明"改价/免单/改余额"必须受控。laundry-desk 至少要覆盖：改价、免单、撤单、删除、改设置、导出。

### P1 — 显著提升柜台效率与经营可控性

1. **暂存挂单**：柜台被打断是高频场景，顺科专门做了确认流程。
2. **库存待取排查 + 催取**：1067 件滞留是这家店真实的钱。至少要能按 收衣日期/未付款/顾客 排查、导出、打印，催取先做"生成名单 + 一键复制手机号"，不必等短信通道。
3. **账目分渠道科目树**：现金/微信/支付宝分列，区分 洗衣消费 / 回收欠款 / 未付欠款 / 撤销。顺科的 `业绩收入 ≠ 真实收入` 口径要照搬。
4. **欠款跨单汇总 + 独立还款入口 + 催付名单**。
5. **打印模板可配**：字段选择、字号、份数、打印时机。顺科的变量清单（21/22 个）可直接作为字段全集参考。
6. **散客档案增强**：车牌、地址、证件、客户标识、多地址；以及客户级开关（不打印取衣单 / 不打印水洗唛）。
7. **备份还原**：门禁项"备份文件可还原到全新安装"当前无法验证，必须补还原侧并做一次演练。
8. **价格资料增强**：助记码检索、单位与内含件数、是否允许打折、参考图片、B 价。顺科的助记码检索在收衣时很关键。
9. **服务大类可配置**：顺科是 11 类且名称可自定义 + F1–F11 快捷键。至少要支持门店自定义大类。

### P2 — 门店成长后再考虑

1. **生产流转（送洗 / 上挂 / 挂点）**：顺科有完整实现但该店零使用。**建议先在真实门店确认是否需要，再决定做不做**——如果店主本来就不外送工厂、不用输送线，这块永远不会被点。
2. **会员卡 / 储值 / 分品类折扣**：1 人办卡的现实说明现在不是时候。若将来做，必须同时实现"业绩收入 ≠ 真实收入"的口径区分。
3. **短信通知**：顺科自己都没开通。laundry-desk 的 `smsService` 已经写完，接线成本低，但价值待验证——先做 P1 的"生成名单 + 复制号码"更划算。
4. **逾期保管费**：顺科有完整规则（免费天数/每件每日/双封顶/会员豁免）但未启用。规则可参考，实现可缓。
5. **上门收送 / 快递**：该店快递账户未开通。
6. **交班结账**：顺科从 2006 年起从未结过一次账。单人店场景下价值低。
7. **员工留言 / 定时提醒**。
8. **业绩分析与客户分层**（消费排名 / 僵尸顾客 / 新增顾客 / 会员分层）：顺科页面全空。

### P3 — 明确不做或最后做

1. **微信公众号 / 线上下单 / 在线支付 / 商城套餐**：顺科未对接，且 laundry-desk 已有 Web + 移动适配这条更现代的路径。
2. **推广员三级分销 / 金币兑换**。
3. **次卡（鞋次卡 / 衣次卡）/ 月月赠 / 团体卡**。
4. **积分抵现 / 邀请返利**。
5. **射频卡 / IC 卡读写器**。
6. **多门店同步**。
7. **管理员高危数据工具**（强制删除 / 批量合并 / 垃圾数据清理）：顺科这块 15 个按钮外观一致、无逐项说明、无二次确认，是**反面教材**。laundry-desk 若将来要做，必须逐项独立确认 + 影响范围预览 + 可撤销。

2. **其次**：备份 + 恢复演练。这是双方共同的空白，也是 laundry-desk 的门禁项。
3. **再次**：权限生效粒度（用"营业员"账号逐个点高危按钮，看拦在哪一层）。
4. **最后**：账目口径对账（用已知的几笔单反推报表数字）。
5. 会员、生产流转、微信、分销**不必花时间实测**——已确认该店零使用，验证收益低。

---

## 6. 一句话结论

顺科在功能面上是一套成熟的洗衣店综合管理系统，但在**这家店**的实际使用面只有六块：收衣开单、三台打印机、取衣、散客档案、欠款回收、库存催取。laundry-desk 的真实差距不在"功能少"，而在**缺少件级衣物实体、件级条码和标签打印**这条物理追踪链，以及**折扣、逾期、短信、审计、还原五处已写但断裂的链路**。会员储值、生产流转、微信生态这些看起来最显眼的部分，证据支持继续不做。
