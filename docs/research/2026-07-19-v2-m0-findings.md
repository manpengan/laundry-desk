## M0-1 RLS 三元租户隔离 + 性能（主责：Codex）

**结论：通过（Codex 自评，最终以 Claude 门禁判定为准）**

**主责 / 完成日期**：Codex / 2026-07-19

**环境**：macOS 26.5.2；Apple M4 Pro（14 核、48 GiB）；Docker client/server 29.3.1；容器 PostgreSQL 16.14（`postgres:16-alpine`，aarch64）。复现说明见 [`tools/spikes/m0-1-rls/README.md`](../../tools/spikes/m0-1-rls/README.md)。

**通过标准逐条对照**：

- [x] 五类旁路读侧全部 0 行：GUC 未设置、空值、事务回滚、同连接切换租户、fresh worker 漏注入均正常返回空集；原始输出见 [`isolation-and-fk.txt`](../../tools/spikes/m0-1-rls/evidence/isolation-and-fk.txt)。
- [x] 写侧 fail-closed：未设置/空 GUC、回滚后残留、worker 漏注入、伪造他租户 INSERT/UPDATE 均报 RLS 违例，无一落库。
- [x] 连接池复用与回滚：同一连接先后执行 tenant A、无注入、tenant B 事务；两次 `SET LOCAL` 在 COMMIT/ROLLBACK 后均无残留。
- [x] 组合外键：同 org 异 store 的 line ID、同 store 异 order 的 line ID 均触发 FK 拒绝。
- [x] 10 万 orders（另有 10 万 lines + 10 万 garments）下三条查询 RLS on P95 为 0.019/0.019/0.022ms，全部 <50ms；三份计划均为复合索引扫描、无 Seq Scan，见 [`benchmark-rls-on.txt`](../../tools/spikes/m0-1-rls/evidence/benchmark-rls-on.txt)。
- [x] RLS on/off 对照已记录；P95 差值为 +0.003/+0.001/+0.004ms，见 [`benchmark-rls-off.txt`](../../tools/spikes/m0-1-rls/evidence/benchmark-rls-off.txt)。
- [x] 业务查询有效角色为 `laundry_app`，`rolbypassrls=false`；owner 仅持有/创建表。角色与 FORCE 状态见 [`schema-introspection.txt`](../../tools/spikes/m0-1-rls/evidence/schema-introspection.txt)。

**关键数据**：

| 查询（每项 250 次） | RLS on P50/P95/P99 (ms) | RLS off P50/P95/P99 (ms) | 命中索引                            |
| ------------------- | ----------------------: | -----------------------: | ----------------------------------- |
| 单店当日单列表      |   0.015 / 0.019 / 0.030 |    0.013 / 0.016 / 0.024 | `orders_store_created_idx`          |
| 按状态过滤          |   0.015 / 0.019 / 0.033 |    0.013 / 0.018 / 0.025 | `orders_store_status_created_idx`   |
| 按客户查单          |   0.017 / 0.022 / 0.036 |    0.015 / 0.018 / 0.030 | `orders_store_customer_created_idx` |

**偏差与坑**：psql 从 stdin 执行时，`\ir` 相对路径按容器工作目录解析；复现脚本因此显式顺序送入 schema、policy、seed。`NULLIF(current_setting(..., true), '')::uuid` 对未设置/空值返回 NULL；非空非法 UUID 会报转换错误，仍 fail-closed，M1 必须只注入已校验的服务端 UUID。

**对设计的影响**：无。实测支持 ADR-02 的 FORCE RLS + 非 owner 应用角色 + 三元组合键方案；未发现需新增 ADR 的阻塞项。

## M0-2 Primary lease 时序 + 可信时间（主责：Codex）

**结论：通过（Codex 自评，最终以 Claude 门禁判定为准）**

**主责 / 完成日期**：Codex / 2026-07-19

**环境**：同 M0-1；Node.js 25.8.2、`pg` 8.22.0。测试 32/32 通过，见 [`test-suite.txt`](../../tools/spikes/m0-2-lease/evidence/test-suite.txt)；逐场景数据见 [`scenarios.jsonl`](../../tools/spikes/m0-2-lease/evidence/scenarios.jsonl)。

**通过标准逐条对照**：

- [x] 双 owner 并发：两个不同 PG backend 连接同时请求，head 行锁排队，结果恰一张 epoch=1 lease；PID 原始值见 [`scenarios.jsonl`](../../tools/spikes/m0-2-lease/evidence/scenarios.jsonl)。唯一 epoch 冲突注入测试证明整事务回滚、无 head 脏写或第二行。
- [x] release ACK 与晋升并发：同一 head 行锁下串行，观测到有效 lease 数量为 1（标准要求 ≤1）。
- [x] 有签名 release ACK：在相同 fake trusted time 立即签发下一张 lease，epoch 从 1 严格增至 2；篡改 ACK 签名验证失败。
- [x] 无 ACK：旧 `not_after=00:00:00.100Z`、skew=10ms；`00:00:00.109Z` 返回 online-only，`00:00:00.110Z` 才签发新 lease。
- [x] 时钟回拨旧主：墙钟回拨被判定为 continuity lost 并永久熔断当前 session，不能续命；新 epoch 生效后，旧 `lease_id+epoch+seq` 回放写入应用角色不可改删的审计、返回 arbitrate，领域回调未执行。
- [x] 六类场景全部 fail-closed：回拨/前跳/进程重启/OS 重启/休眠恢复/旧主失联分别返回 continuity/identity/deadline 失败，统一降级 online-only；另证请求途中 sleep/reboot 也无法启用 lease。
- [x] RTT≥TTL：RTT=100ms、TTL=100ms 拒绝启用；RTT=85ms、margin=10ms 的 `local_deadline=90ms`，不晚于 100ms 服务端权力窗口。
- [x] `src/` 有效性路径静态检查无 `Date.now()`，见 [`date-now-static-check.txt`](../../tools/spikes/m0-2-lease/evidence/date-now-static-check.txt)；截止只用 request-start monotonic anchor。

**关键数据**：

| 场景                   | 期望                            | 实测                                     |
| ---------------------- | ------------------------------- | ---------------------------------------- |
| 双 owner 并发          | 恰一张 lease                    | 1 issued，2 个 backend PID               |
| 无 ACK 等待            | `not_after+skew` 前 online-only | 109ms 拒绝，110ms 签发                   |
| release / promote 竞态 | 同时有效 ≤1                     | 1                                        |
| RTT=TTL                | fail-closed                     | `rtt-at-least-ttl`                       |
| 旧 epoch 回放          | 审计 + 仲裁 + 不应用            | `arbitrate`，`arbitration_required=true` |
| 唯一约束冲突           | 全事务回滚                      | lease 行仍 1，head 保持冲突前快照        |

签名 lease 样例（含 `issued_at/ttl_ms/max_clock_skew_ms/not_after`）在 [`scenarios.jsonl`](../../tools/spikes/m0-2-lease/evidence/scenarios.jsonl) 的 `signed-lease-sample` 行；PG 可信时间、行锁和单调截止的静态位置见 [`critical-path-static-check.txt`](../../tools/spikes/m0-2-lease/evidence/critical-path-static-check.txt)。

**偏差与坑**：fake clock 仅为测试接缝；未注入时实现从 PostgreSQL `clock_timestamp()` 读取可信时间。墙钟只用于识别不可证明的跳变，不参与到期计算。Edge 必须先按版本化 canonical message 验签，并在请求发出前及响应到达时各取 monotonic + boot/process/wake/wall 快照；缺失、非有限或请求途中变化均 fail-closed。授权函数是不可变状态机，调用方必须原子保存返回的新 session，生产封装不得复用旧快照。release 持有 head 锁期间的设备公钥解析必须是同步内存/事务内操作，不能等待外部 I/O。回放必须维护每 lease 高水位；乱序、精确重复、同 seq 内容碰撞及已 release lease 均不能再次触发领域写入。

**对设计的影响**：无。实测支持 ADR-04 的 head 行锁、无 ACK 等待、请求前单调锚点与旧 epoch 仲裁语义；未发现需新增 ADR 的阻塞项。

## M0-3 三类打印机（Grok）

### 产出位置
